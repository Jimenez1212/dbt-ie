## What is Analytics Engineering?

A discipline that applies **software engineering principles** to data transformation.

- **The Practices**: Version control (Git), CI/CD, Automated Testing, Modular Code.
- **The Gap**: Bridges the gap between Data Engineering (Infrastructure/Ingestion) and Data Analysis (Insights/Reporting).
- **The Outcome**: Trusted, documented, and reliable datasets that the business can depend on.

---

## What is dbt?

**dbt** is the transformation workflow that lets you quickly and collaboratively deploy analytics code.

- **The "T" in ELT**: dbt doesn't extract or load data. It transforms data already in your warehouse.
- **SQL-First**: Allows anyone who knows SQL to build production-grade data pipelines.
- **Compiler**: Compiles Jinja (templating) and SQL into raw SQL to run against your database (DuckDB, Snowflake, etc.).

---

## The Rise of dbt

dbt has emerged as the **de-facto standard** for data transformation in the Modern Data Stack.

- **Community**: Thousands of professionals in the [dbt Slack community](https://www.getdbt.com/community/join-the-community)
- **Adoption**: Used by thousands of companies, from startups to Fortune 500s.
- **Ecosystem**: Central to the modern stack, integrating with tools like Fivetran, Airflow, and all major data warehouses.
- **Career**: "Analytics Engineer" is one of the fastest-growing roles in data.

## The Role of Analytics Engineer

**Responsibilities**:
- **Data Modeling**: Designing clean, modular data structures (Marts).
- **Software Engineering**: Applying version control, testing, and CI/CD to data.
- **Documentation**: Ensuring data is self-describing and trustworthy.

**Skills**:
- **SQL & Python**: Fluency required.
- **dbt**: Proficiency in the tool.
- **Communication**: Translating business needs into technical requirements.

**Position**: Sits right between Data Engineering and Data Analysis.

## Introduction to DuckDB

**Why DuckDB?**

- **"SQLite for Analytics"**: An in-process SQL OLAP database management system.
- **Serverless**: No need to install or manage a database server.
- **Fast**: Optimized for analytical queries (columnar storage).
- **Integration**: Works seamlessly with Python, dbt, and Parquet files.

In this course, DuckDB acts as our Data Warehouse, running locally on your machine.

## The `dbt_project.yml` File

The heart of your dbt project.

- **Project Identity**: Name, version, profile.
- **Model Configurations**: Materializations, tags, schemas (applied globally or by folder).
- **Paths**: Where dbt looks for models, seeds, tests, etc.

```yaml
name: 'my_project'
version: '1.0.0'
profile: 'my_profile'

models:
  my_project:
    staging:
      +materialized: view
```

---

## Project Folder Structure

- `models/`: Your SQL transformations.
- `seeds/`: CSV files for static data (country codes, mappings).
- `tests/`: Custom SQL tests.
- `analyses/`: Ad-hoc queries (compiled but not run).
- `macros/`: Reusable Jinja functions.
- `snapshots/`: SCD Type 2 logic.

## The Compilation Process

dbt is a compiler. It takes your code (Jinja + SQL) and turns it into raw SQL.

1.  **Parse**: Reads all files and builds a graph.
2.  **Compile**: Resolves `{{ ref() }}`, `{{ source() }}`, and variables.
3.  **Execute**: Wraps the SQL in DDL (`CREATE VIEW/TABLE`) and runs it.

Check `target/compiled/` to see the SQL dbt generates.
Check `target/run/` to see the DDL dbt executes.

---

## Compilation Example

**Source Code (`models/my_model.sql`)**:
```sql
select *
from {{ ref('stg_customers') }}
where created_at > '{{ var("start_date") }}'
```

**Compiled Code (`target/compiled/.../my_model.sql`)**:
```sql
select *
from "analytics"."dbt_dgarcia"."stg_customers"
where created_at > '2023-01-01'
```

dbt resolves the `ref()` to the actual database object and substitutes variables.

---

## The Target Directory

The `target/` folder is the "build artifact" directory. It is **git-ignored** and generated by dbt commands.

- **`compiled/`**: Contains the raw `SELECT` statements (Jinja resolved).
- **`run/`**: Contains the actual DDL executed (`CREATE VIEW/TABLE ...`).
- **`manifest.json`**: The full state of your project (used for state comparison).
- **`run_results.json`**: Results of the last run (success/fail, timing).
- **`catalog.json`**: Metadata from the database (only generated by `dbt docs generate`).

*Tip: Always check `target/compiled` when debugging logic!*

## Configuring Model Access (Intro)

Control how models can be referenced.

- **Public**: Can be referenced by any project.
- **Protected**: Can be referenced within the project or by specific upstream projects.
- **Private**: Can only be referenced within the same group/folder.

```yaml
models:
  +access: protected
```
